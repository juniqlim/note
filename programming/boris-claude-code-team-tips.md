# Claude Code 팀 내부 사용 팁
https://x.com/bcherny/status/2017742741636321619

> Claude Code의 제작자인 Boris가 Claude Code 팀 내부에서 사용하는 팁을 공유한 내용의 한국어 번역입니다.

저는 Boris이고, Claude Code를 만들었습니다. Claude Code 팀으로부터 직접 전해 들은 몇 가지 사용 팁을 빠르게 공유하고자 합니다. 팀원들이 Claude를 사용하는 방식은 제가 사용하는 방식과는 또 다릅니다. 기억하세요: Claude Code를 사용하는 데 있어 단 하나의 정해진 정답은 없습니다. 모든 사람의 설정은 다릅니다. 자신에게 맞는 방법을 찾기 위해 실험해 보세요!

---

## 1. 병렬로 더 많은 작업 처리하기

한 번에 3~5개의 **Git 워크트리(worktree)**를 띄우고, 각 워크트리에서 별도의 Claude 세션을 병렬로 실행하세요. 이것은 생산성을 가장 크게 높여주는 방법이며, 팀이 꼽는 최고의 팁입니다. 개인적으로 저는 여러 개의 git checkout을 사용하지만, Claude Code 팀 대부분은 워크트리를 선호합니다. 이것이 바로 @amorriscode가 Claude Desktop 앱에 워크트리 지원 기능을 내장한 이유이기도 합니다!

어떤 사람들은 워크트리에 이름을 붙이고 쉘 별칭(alias: za, zb, zc 등)을 설정하여 키보드 한 번으로 워크트리 간을 이동합니다. 또 다른 사람들은 로그를 읽거나 BigQuery를 실행하는 용도로만 사용하는 전용 "분석(analysis)" 워크트리를 두기도 합니다.

---

## 2. 복잡한 작업은 항상 '계획(Plan)' 모드로 시작하기

구현 단계에서 Claude가 한 번에(1-shot) 해결할 수 있도록, 계획 수립에 에너지를 쏟으세요.

어떤 사람은 하나의 Claude에게 계획을 작성하게 한 뒤, 두 번째 Claude를 실행해 '스태프 엔지니어(Staff Engineer)' 관점에서 그 계획을 검토하게 합니다.

다른 팀원은 일이 잘못된 방향으로 가기 시작하면 즉시 '계획 모드'로 돌아가 다시 계획을 세웁니다. 무리해서 계속 진행하지 않습니다. 또한 빌드뿐만 아니라 검증 단계에서도 Claude에게 명시적으로 계획 모드로 진입하라고 지시합니다.

---

## 3. CLAUDE.md에 투자하기

모든 수정 작업이 끝난 후에는 항상 이렇게 마무리하세요: "CLAUDE.md를 업데이트해서 다시는 같은 실수를 하지 않도록 해." Claude는 자신을 위한 규칙을 작성하는 데 무서울 정도로 능숙합니다. 시간을 들여 CLAUDE.md를 가차 없이 수정하세요. Claude의 실수 비율이 눈에 띄게 떨어질 때까지 반복해서 다듬으세요. 한 엔지니어는 Claude에게 모든 작업/프로젝트에 대해 노트 디렉토리를 유지하게 하고, PR이 끝날 때마다 업데이트하게 합니다. 그리고 CLAUDE.md가 그 노트를 참조하도록 설정합니다.

---

## 4. 나만의 스킬을 만들어 Git에 커밋하기

모든 프로젝트에서 재사용할 수 있도록 하세요. 팀의 팁:

하루에 두 번 이상 하는 작업이라면 스킬이나 명령어로 만드세요.

/techdebt 슬래시 커맨드를 만들어 매 세션이 끝날 때마다 실행하여 중복된 코드를 찾아 제거하세요.

Slack, GDrive, Asana, GitHub의 최근 7일 치 데이터를 한 번에 컨텍스트로 덤프(dump)하는 슬래시 커맨드를 설정하세요.

dbt 모델을 작성하고, 코드를 리뷰하고, 개발 환경에서 변경 사항을 테스트하는 '분석 엔지니어 스타일'의 에이전트를 구축하세요.

---

## 5. 대부분의 버그는 Claude가 스스로 수정합니다

저희가 하는 방법은 다음과 같습니다:

Slack MCP를 활성화한 다음, Slack의 버그 스레드를 Claude에 붙여넣고 그냥 **"고쳐줘(fix)"**라고 말하세요. 컨텍스트 스위칭이 전혀 필요 없습니다.

혹은 그냥 "실패하는 CI 테스트를 고쳐줘"라고 하세요. 어떻게 할지 세세하게 간섭(micromanage)하지 마세요.

분산 시스템 문제를 해결할 때는 Claude에게 Docker 로그를 보여주세요. 이 작업에 놀라울 정도로 능숙합니다.

## 6. 프롬프팅 레벨 높이기

**a. Claude에게 도전하세요:** "이 변경 사항들에 대해 나를 깐깐하게 검증(Grill me)해보고, 네 테스트를 통과하기 전까진 PR을 만들지 마."라고 말하세요. Claude를 리뷰어(reviewer)로 활용하세요. 혹은 "이게 작동한다는 걸 증명해 봐"라고 하고 메인 브랜치와 기능 브랜치 간의 동작 차이(diff)를 보여달라고 하세요.

**b. 수정 결과가 별로라면:** "네가 지금 알고 있는 모든 지식을 동원해서, 이걸 다 갈아엎고(scrap) 우아한 해결책으로 다시 구현해 봐."라고 하세요.

**c. 작업을 넘기기 전에 상세한 명세를 작성하고 모호함을 줄이세요.** 구체적일수록 결과물은 더 좋아집니다.

## 7. 터미널 및 환경 설정

팀원들은 Ghostty를 정말 좋아합니다! 동기화된 렌더링, 24비트 컬러, 완벽한 유니코드 지원 때문에 많은 사람이 애용합니다. Claude를 더 쉽게 다루기 위해, /statusline을 사용하여 상태 표시줄에 항상 컨텍스트 사용량과 현재 git 브랜치가 표시되도록 설정하세요. 우리 중 다수는 터미널 탭에 색상을 지정하고 이름을 붙이며, 때로는 tmux를 사용하여 작업/워크트리당 하나의 탭을 사용합니다. 음성 받아쓰기를 사용하세요. 타이핑하는 것보다 3배 빠르며, 결과적으로 프롬프트가 훨씬 더 구체적이게 됩니다. (macOS에서는 fn 키 두 번)

## 8. 서브 에이전트(Subagents) 사용하기

**a.** Claude가 문제 해결에 더 많은 연산(compute)을 투입하길 원한다면 요청 뒤에 "use subagents"를 붙이세요.

**b.** 메인 에이전트의 컨텍스트 윈도우를 깨끗하고 집중된 상태로 유지하기 위해 개별 작업은 서브 에이전트에 위임하세요.

**c.** 훅(hook)을 통해 권한 요청을 Opus 4.5로 라우팅하세요. 공격 여부를 스캔하고 안전한 요청은 자동으로 승인하게 할 수 있습니다.

---

## 9. 데이터 및 분석에 Claude 활용하기

Claude Code에게 "bq" CLI를 사용하여 즉석에서 지표를 추출하고 분석하도록 요청하세요. 우리는 코드베이스에 BigQuery 스킬을 체크인(포함)해 두었으며, 팀의 모든 사람이 Claude Code 내에서 직접 분석 쿼리를 실행하는 데 사용합니다. 개인적으로 저는 6개월 넘게 SQL을 한 줄도 작성하지 않았습니다. 이 방법은 CLI, MCP, 또는 API가 있는 모든 데이터베이스에서 작동합니다.

---

## 10. Claude와 함께 학습하기

Claude Code를 학습용으로 활용하는 팀의 팁입니다:

**a.** /config에서 "Explanatory(설명형)" 또는 "Learning(학습형)" 출력 스타일을 활성화하여 Claude가 변경 사항에 대한 *이유(why)*를 설명하게 하세요.

**b.** 낯선 코드를 설명하는 시각적인 HTML 프레젠테이션을 생성해 달라고 하세요. 놀라울 정도로 훌륭한 슬라이드를 만들어줍니다!

**c.** 새로운 프로토콜이나 코드베이스를 이해하기 위해 ASCII 다이어그램을 그려달라고 요청하세요.

**d.** 간격 반복(spaced-repetition) 학습 스킬을 만드세요: 사용자가 이해한 내용을 설명하면, Claude가 후속 질문을 통해 빈틈을 채워주고 그 결과를 저장하게 하세요.
